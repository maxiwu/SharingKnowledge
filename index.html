Engineering Proposal ENG-PROP-2026-001

Stop Treating Symptoms.
Fix the Root.

A structured workflow to identify, safely refactor, and permanently resolve recurring bugs caused by stale state and poor design — rather than patching them indefinitely.

01 — The Problem

We Are Paying a Recurring Tax

Every sprint, engineering time is spent fixing the same class of bugs — not because developers make careless mistakes, but because the underlying design creates conditions where bugs are inevitable. Fixing symptoms without fixing the cause means this tax never ends.

The gap between stale in-memory state and the source of truth is not a bug. It is an architectural condition that silently produces bugs — until the design is corrected.

Root Cause

Entity data is read from persistent storage and duplicated in memory. When the source of truth is mutated, the duplicate is not updated atomically — creating a window of incorrect state.

Hidden Danger

Developers unaware of the duplication read from the stale copy. They get missing data or wrong data with no warning — producing incorrect summaries, failed validations, and silent data errors.

Why It Recurs

Each fix patches the symptom at the callsite. The underlying duplication remains. The next developer, the next feature, and the next module repeat the same mistake.

Business Impact

Incorrect test summary and validation results. Bugs reaching production. Repeated hotfixes consuming sprint capacity that should go to product value.

02 — The Workflow

Five Steps from Symptom to Resolution

Each step has a defined entry condition, output, and — where decisions carry risk — a sign-off gate. No step is skipped; no refactoring begins without a safety net in place.

01
Root Cause Analysis & Postmortem

A senior engineer reviews the bug against the RCA trigger checklist. If any criterion is met, a formal RCA is opened. The goal is to confirm a recurring tax exists and identify the structural root cause — not the proximate bug.

Senior Engineer Review RCA Trigger Checklist Output: Root Cause Statement
02
Risk Evaluation & Safety Net

Assess the scope and blast radius of the required refactoring. Build characterization tests that capture current behavior — including known-buggy behavior — and regression tests for critical paths. No refactoring begins until the safety net is in place.

Characterization Tests Regression Tests Output: Test Coverage Report
ADR
Architecture Decision Record — Sign-off Gate

The proposed solution is documented as an ADR covering scope, method, trade-offs, and feasibility. The code owner or a senior member with module knowledge must approve before design work begins. The ADR progresses through three explicit states.

Draft Author writes
→
Approved Code owner signs
→
Implemented Work begins
Approval gate — work cannot proceed without sign-off
03
Solution Design & Feasibility Spike

Design the refactoring solution — defining the target state, the migration path, and the scope boundary. Run a time-boxed spike to validate that the solution is both technically feasible and demonstrably effective before full implementation is committed.

Time-boxed Spike Scope Definition Output: Validated Design
04
Implementation

Implement the change within the agreed scope. All characterization and regression tests must pass. Where the correct new design intentionally changes expected behavior, a domain expert from the business team reviews and signs off before the test baseline is updated — treated the same as a mutation test result.

Must Pass: Characterization Tests Must Pass: Regression Tests Domain Expert Sign-off on Intentional Changes
05
Delivery & Knowledge Archive

The change is delivered. The ADR is marked Implemented and archived. The change document — including root cause, design decision, and test rationale — is stored with the codebase so future developers understand why the design exists, not just what it does.

ADR Archived Change Document Knowledge Preserved
03 — The Trigger

RCA Trigger Checklist

A senior engineer evaluates the bug against this checklist. Any single criterion is sufficient to open a formal RCA. The checklist prevents both under-reaction (ignoring systemic bugs) and over-reaction (RCA on every trivial fix).

Pattern Signals

  • Same bug pattern appeared more than once, even across different modules
  • A previous fix for this bug was later reverted or re-patched
  • This module is in the top 3 by total bug count
  • The module's new bug rate is increasing sprint over sprint

Severity Signals

  • Bug reached production — not caught in QA or staging
  • Bug caused incorrect business output: wrong summary, wrong validation, wrong decision
  • Bug caused data corruption or data loss

Complexity Signals

  • Fix required touching more than one module or ~5+ files
  • Time to diagnose exceeded half a day
  • The developer who fixed it could not explain why it happened

One-hit Threshold

  • Any single item from any category above is sufficient to trigger
  • Checklist is evaluated by a senior engineer — not self-assessed by the fixing developer
  • Output of trigger is a Root Cause Statement, not a fix
04 — Risk Mitigation

Rollback Strategy

Refactoring without a rollback plan is a liability. We use a tiered approach — the minimum viable recovery for most cases, with an escalation path for high-risk modules.

Tier 1 — Default
Git Revert

All changes are committed with clear scope boundaries. If a regression is detected, the commit is reverted immediately. Zero code archaeology required.

Tier 2 — High Risk
Train Track

Both old and new implementations coexist. Callsites are migrated incrementally. A strategy pattern switch allows instant fallback to the old implementation at any time.

Exit Criteria
Remove Old Code

The old implementation is removed after one full release cycle — typically a quarter — with zero incidents reported in the field. Removal task is created at implementation time.

05 — The Ask

What We Need to Start

This workflow does not require a large upfront investment. It requires agreement on the process, and authorization to apply it to the first confirmed case — which we already have evidence for.

Process Approval

Adopt this workflow as the standard response to recurring bugs. No ad-hoc patches for systemic issues.

Senior Engineer Time

Designate a senior engineer or code owner per module as the RCA trigger evaluator and ADR approver.

Domain Expert Access

When characterization test baselines need to change, a business-side domain expert must be available to validate expected outcomes.

Resource Escalation Path

When domain expert or code owner availability blocks progress, the manager is the escalation point to resolve the resource constraint.

This proposal addresses the reactive phase of the problem. A companion proposal covering proactive prevention — design guidelines, review checklists, and standards for new code — will follow separately.
